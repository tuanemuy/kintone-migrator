# 利用シナリオ詳細

## 前提

- UIはレコード一覧画面のヘッダースペースに表示する
- ヘッダースペースは横幅が限られるため、操作のエントリーポイントはコンパクトにまとめる
- メイン操作はドロップダウンメニューでまとめ、「フォーム管理」ボタンから展開する
- 各操作の詳細表示や確認にはダイアログ（モーダル）を使用する
    - ダイアログを選択する理由: レコード一覧画面のコンテキストを維持しつつ、操作に集中させるため
    - ページ遷移は行わない（kintoneカスタマイズの制約上、同一画面内で完結させる）

## 画面構成

```
[レコード一覧画面のヘッダースペース]
┌─────────────────────────────────────┐
│  [フォーム管理 ▼]                     │
│    ├ 差分を確認する                    │
│    ├ マイグレーションを実行する          │
│    ├ 強制的に上書きする                 │
│    └ 現在の設定を取り込む               │
└─────────────────────────────────────┘
```

## シナリオ1: スキーマファイルの宣言的設定と現在のフォーム設定の差分を確認する

> 注: 本シナリオ群（シナリオ1〜4）の「スキーマ（スキーマファイル）」は FormSchema ドメインが扱う宣言的なフォーム構成であり、`SchemaStorage`/`SchemaParser` 経由で永続化・パースされる（[FormSchema ユースケース](../usecases/formSchema.md) 参照）。kintone の「アプリ管理者用メモ（AdminNotes）」機能とは別概念であり、混同しないこと。

### 目的

フォーム設定を変更する前に、スキーマファイルに記載した宣言的設定と実際のフォーム設定の間にどのような差分があるかを把握する。

### フロー

1. 利用者はレコード一覧画面を開く
2. ヘッダースペースの「フォーム管理」ドロップダウンメニューから「差分を確認する」を選択する
3. システムは以下を並行して取得する
    - スキーマファイルに記載された宣言的設定をパースする
    - kintone REST APIで現在のフォームフィールド設定を取得する
4. システムは両者を比較し、差分を検出する
5. **ダイアログ**が開き、差分の一覧を表示する

### ダイアログの表示内容

- タイトル: 「フォーム設定の差分」
- 差分がある場合:
    - 変更種別ごとにセクションを分けて表示する
        - **追加**: 宣言的設定にあるが、現在のフォームにないフィールド
        - **変更**: 両方に存在するが設定値が異なるフィールド
        - **削除**: 現在のフォームにあるが、宣言的設定にないフィールド
    - 各フィールドについて以下を表示する
        - フィールドコード
        - フィールド名
        - 変更内容の要約（例: 「必須 → 任意」「新規追加」など）
    - 変更件数のサマリーを上部に表示する（例: 追加 2件, 変更 3件, 削除 1件）
- 差分がない場合:
    - 「差分はありません。フォーム設定は宣言的設定と一致しています。」と表示する
- フッター: 「閉じる」ボタン

### エラー時

- スキーマファイルのパースに失敗した場合: ダイアログ内にエラーメッセージを表示する（例: 「スキーマファイルの設定を読み取れませんでした。記述形式を確認してください。」）
- API取得に失敗した場合: ダイアログ内にエラーメッセージを表示する（例: 「フォーム設定の取得に失敗しました。」）

## シナリオ2: マイグレーションを実行してフォーム設定を更新する

### 目的

スキーマファイルの宣言的設定と現在のフォーム設定の差分を確認した上で、差分のみを適用してフォーム設定を更新する。

### フロー

1. 利用者はレコード一覧画面を開く
2. ヘッダースペースの「フォーム管理」ドロップダウンメニューから「マイグレーションを実行する」を選択する
3. システムは差分を検出する（シナリオ1と同様の処理）
4. **確認ダイアログ**が開き、適用される変更の一覧を表示する

### 確認ダイアログの表示内容

- タイトル: 「マイグレーションの確認」
- 差分がある場合:
    - シナリオ1と同様の差分一覧を表示する
    - 注意書き: 「以下の変更をフォーム設定に適用します。」
    - フッター:
        - 「キャンセル」ボタン（ダイアログを閉じる）
        - 「実行する」ボタン（プライマリ、マイグレーションを実行する）
- 差分がない場合:
    - 「差分はありません。マイグレーションは不要です。」と表示する
    - フッター: 「閉じる」ボタン

### 実行中の表示

- 「実行する」ボタンを押すと、ボタンがローディング状態になる
- ダイアログは開いたまま、操作を受け付けない状態にする

### 実行完了後

- 成功時: ダイアログを閉じ、トースト通知で「マイグレーションが完了しました。」と表示する
- 失敗時: ダイアログ内にエラーメッセージを表示する（例: 「マイグレーションに失敗しました: フィールド○○の追加でエラーが発生しました。」）。ダイアログは閉じずに、利用者がエラー内容を確認できるようにする

## シナリオ3: スキーマファイルの設定でフォーム設定を強制的に上書きする

### 目的

差分の有無にかかわらず、スキーマファイルの宣言的設定でフォーム設定を完全に上書きする。宣言的設定に含まれないフィールドは削除される。

### フロー

1. 利用者はレコード一覧画面を開く
2. ヘッダースペースの「フォーム管理」ドロップダウンメニューから「強制的に上書きする」を選択する
3. システムは現在のフォーム設定と宣言的設定の両方を取得する
4. **警告ダイアログ（AlertDialog）** が開き、上書き内容を表示する

### 警告ダイアログの表示内容

- タイトル: 「フォーム設定の強制上書き」
- 警告アイコンと警告メッセージ: 「この操作はフォーム設定を宣言的設定で完全に上書きします。宣言的設定に含まれないフィールドは削除されます。この操作は元に戻せません。」
- 上書き後のフィールド一覧を表示する
- 削除されるフィールドがある場合は強調表示する（destructiveスタイル）
- フッター:
    - 「キャンセル」ボタン（ダイアログを閉じる）
    - 「上書きする」ボタン（destructiveスタイル、上書きを実行する）

### 実行中の表示

- シナリオ2と同様にローディング状態にする

### 実行完了後

- 成功時: ダイアログを閉じ、トースト通知で「フォーム設定を上書きしました。」と表示する
- 失敗時: ダイアログ内にエラーメッセージを表示する。ダイアログは閉じない

## シナリオ3a: フォーム設定をリセットして全カスタムフィールドを削除する

### 目的

フォームからすべてのカスタムフィールドを削除し、空の初期状態に戻す。テスト環境のクリーンアップや、フォーム設定をゼロから作り直す際に使用する。CLIでは `override --reset` フラグで実行する。

### 前提

- `--reset` と `--schema` フラグは排他的であり、同時に指定した場合は `ValidationError` となる
- スキーマファイルの指定は不要

### フロー（CLI）

1. 利用者は `override --reset` を実行する
2. システムは `FormConfigurator.getFields()` で現在のフォーム設定を取得する
3. システムはシステムフィールドを除外し、削除対象のカスタムフィールドを特定する
4. 削除対象フィールドの一覧を表示し、確認プロンプトを表示する
    - 「全 N 件のカスタムフィールドを削除してフォームをリセットします。この操作は元に戻せません。続行しますか？ (y/N)」
5. 利用者が確認した場合、`FormConfigurator.deleteFields()` で全カスタムフィールドを削除する
6. `FormConfigurator.updateLayout()` で空のレイアウトに更新する

### 出力例

```
Reset target: 12 custom fields will be deleted.

Fields to delete:
  - customer_name (SINGLE_LINE_TEXT)
  - customer_code (SINGLE_LINE_TEXT)
  - email (SINGLE_LINE_TEXT)
  ...

⚠ All 12 custom fields will be deleted and the form will be reset. This cannot be undone.
Continue? (y/N): y

Deleting fields... done.
Updating layout... done.

Form has been reset successfully.
```

### カスタムフィールドがない場合

```
No custom fields found. The form is already in its initial state.
```

### Multi-Appモード

- `override --reset --all` で全アプリのフォームを一括リセットできる
- 依存関係の逆順（依存されているアプリを後に）でリセットを実行する
    - 理由: 参照先アプリのフィールドを先に削除すると、参照元のLookupフィールド等が不整合になる可能性があるため
- 1回の確認プロンプトで全アプリの実行を開始する
- Fail-Fast方式: 1つのアプリで失敗した場合、残りのアプリはスキップされる

### エラー時

- `--reset` と `--schema` を同時に指定した場合: `ValidationError`（「--reset と --schema は同時に指定できません。」）
- `FormConfigurator.getFields()` の通信に失敗した場合: `SystemError` を表示する
- `FormConfigurator.deleteFields()` の通信に失敗した場合: `SystemError` を表示し、処理を中断する

## シナリオ4: 稼働中のフォーム設定から宣言的な構成を生成してスキーマファイルに反映する

### 目的

既にフォーム設定が存在するアプリに対して、現在の設定を宣言的な構成としてスキーマファイルに書き出す。初回導入時や、手動で変更した設定を宣言的管理に取り込む際に使用する。

### フロー

5. **確認ダイアログ**が開き、生成された構成のプレビューを表示する

### 確認ダイアログの表示内容

- タイトル: 「設定の取り込み」
- 生成された宣言的設定のプレビューを表示する（スクロール可能な領域に表示）
- スキーマファイルに既に設定が存在する場合は注意書きを表示する: 「スキーマファイルの既存の設定は上書きされます。」
- フッター:
    - 「キャンセル」ボタン（ダイアログを閉じる）

- 失敗時: ダイアログ内にエラーメッセージを表示する。ダイアログは閉じない

## シナリオ5: 複数アプリを依存順に一括操作する（Multi-Appモード）

### 目的

プロジェクト設定ファイル（`kintone-migrator.yaml`）で定義された複数のkintoneアプリに対して、依存関係に基づいた順序で一括操作（diff/migrate/override/capture/dump）を実行する。

### 前提

- `kintone-migrator.yaml` がプロジェクトルートに配置されている
- CLI引数 `--all` で全アプリ一括実行、`--app <name>` で単一アプリ実行

### フロー（`--all` での一括 migrate の例）

1. 設定ファイルを読み込み、バリデーションを実行する
2. 依存関係を解決し、トポロジカルソートで実行順序を決定する
3. 各アプリのdiffを依存順に検出して表示する
4. 変更サマリーを表示し、1回の確認プロンプトで続行を問う
5. 利用者が確認した場合、依存順に各アプリのmigration + deployを実行する
6. 失敗時は即停止し、残りのアプリはスキップする

### 出力例

```
=== [customer] (app: 10) ===
Migration + Deploy completed.

=== [order] (app: 20) ===
[SystemError] EXTERNAL_API_ERROR: Failed to add form fields

Execution stopped.
  ✓ Succeeded: customer
  ✗ Failed: order
  - Skipped: invoice
```

### エラー時

- 設定ファイルが見つからない場合: ValidationErrorを表示
- 循環依存が検出された場合: BusinessRuleErrorを表示
- 実行中のアプリが失敗した場合: Fail-Fastで即停止、結果サマリーを表示

## 横断ルール（全シナリオ共通）

FormSchema 管理（シナリオ1〜4）と Multi-App 管理（シナリオ5）以外の設定管理シナリオは、大半が共通の apply / capture / diff パターンに従う。本節で横断ルールを定め、続く「共通シナリオテンプレート」と「設定種別ごとの差分一覧」で網羅性を担保する。

### 提供チャネルの線引き（UI / CLI / 両方）

各シナリオがどのチャネルで提供されるかを以下に整理する。

| シナリオ群 | UI（レコード一覧ヘッダー） | CLI | 備考 |
| --- | --- | --- | --- |
| フォームスキーマ管理（シナリオ1〜4） | 提供 | 提供 | UIはダイアログ完結。CLIは `schema diff` / `schema migrate` / `schema override` / `schema capture` 等 |
| フォームリセット（シナリオ3a） | 非提供 | 提供 | `schema override --reset`。UIには対応操作を置かない |
| その他すべての設定管理（シード・カスタマイズ・各種ACL・ビュー・プロセス管理・一般設定・通知・レポート・アクション・管理者用メモ・プラグイン） | 非提供 | 提供 | CLI中心。`<domain> apply` / `<domain> capture`（+ `diff` を持つドメインあり） |
| Multi-App 一括操作（シナリオ5） | 非提供 | 提供 | `--all` / `--app <name>` |

- UIシナリオ（FormSchema）の判断・確認はダイアログで完結し、ページ遷移を行わない。
- CLIシナリオの確認は標準入力のプロンプト（`y/N`）で行う。
- 以降の共通テンプレートは「CLI中心の設定管理シナリオ」を前提に記述する。UI固有の挙動が必要な場合はシナリオ1〜4の記述を参照する。

### 冪等性

- **apply**: 同じ設定ファイルで apply を再実行しても結果は変わらない（安全に再実行可能）。全置換・マージ・部分更新・Upsert のいずれの反映方式でも、入力が同じであれば最終状態は同じになる。マージ方式（プラグイン・カスタマイズ）では既に反映済みの項目は重複追加されず、no-op となる。
- **capture**: 同一のリモート状態に対して capture を再実行すると、同一内容の設定ファイル（ラウンドトリップ整合）が生成される。
- **diff**: 副作用を持たず、何度実行してもリモート・ローカルの状態を変更しない。
- 反映方式ごとの冪等性の差は「設定種別ごとの差分一覧」の「反映方式」列を参照する。

### 破壊的操作と確認フロー

全置換・削除・リセットを伴う操作は、実行前に必ず確認を挟む。

- **CLI**: 破壊的変更（全置換による既存項目の削除、`--clean`、`override`、`override --reset` 等）の前に削除・上書き対象のサマリーを表示し、`y/N` プロンプトで明示同意を求める。同意がなければ中断する。
- **UI（FormSchema）**: 強制上書き・リセット相当は警告ダイアログ（AlertDialog）で行い、削除されるフィールドを destructive スタイルで強調し、実行ボタンも destructive スタイルにする。
- **全置換系**（ACL各種・ビュー・プロセス管理・レポート・アクション）: 「設定ファイルに存在しない項目はリモートから削除される」旨を確認時に明示する。
- **Multi-App**: 全アプリ分の変更を集約したうえで、1回の確認プロンプトで一括実行を開始する。

### capture / dump の上書き保護

- `capture` / `dump` 系は既存の出力ファイルが存在する場合、上書き警告を表示する。
- `--force` フラグを指定した場合は警告を抑止して上書きする。未指定で既存ファイルがある場合は、確認なしに上書きせず警告・中断を基本とする。
- Multi-App の `capture` / `dump` は各アプリの設定済みパス（`<domain>/<appName>.yaml` 等）に出力し、同じ上書き保護を個別に適用する。

### 異常系チェックリスト（6エラー種別基準）

各シナリオでは、CLAUDE.md が定義する6エラー種別を基準に以下を想定する。UI系シナリオで従来扱いの薄かった認証・権限・競合・デプロイ未完了も明示的に補う。

| エラー種別 | 代表的な発生条件 | 想定対応 |
| --- | --- | --- |
| `ValidationError` | 設定ファイルの記述形式不正・必須項目欠落・排他フラグの同時指定・設定ファイル/スキーマファイル不在 | 該当箇所を示すメッセージを表示し中断 |
| `NotFoundError` | 対象アプリ・対象リソースが存在しない | 対象を示して中断 |
| `UnauthenticatedError` | 認証情報の不備・トークン/パスワード誤り（401相当） | 認証設定の確認を促し中断 |
| `ForbiddenError` | 対象アプリ・設定領域への操作権限不足（403相当） | 権限不足を示して中断 |
| `ConflictError` | revision 競合（取得から更新までの間にリモートが変更された）・楽観ロック失敗 | 競合を示し、再取得・再実行を促す |
| `SystemError` | kintone REST API 通信失敗・ファイルIO失敗・デプロイ未完了/失敗 | 失敗箇所を示して中断（apply途中失敗時は後続操作を実行しない） |

- **revision 競合（ConflictError）**: revision（楽観ロック）を扱うドメイン（appPermission/recordPermission/view/processManagement/generalSettings/notification/report/action/adminNotes/customization 等）の apply は、capture/get 時に取得した revision を update に渡す。リモートが先に変更されていれば競合として検出される。
- **デプロイ未完了**: apply 後の運用環境反映（デプロイ）が未完了・失敗した場合は `SystemError` として扱い、変更が本番に反映されていない可能性を示す。

## 共通シナリオテンプレート（apply / capture / diff）

設定管理シナリオの大半は以下の標準フローに従う。各設定種別の固有差分は「設定種別ごとの差分一覧」を参照する。

### apply 系（`<domain> apply`）標準フロー

1. **前提条件**: 認証情報（接続先・APIトークンまたはユーザー名/パスワード）と対象アプリID（`KINTONE_APP_ID` または設定ファイル）が解決済みであること。設定ファイルが指定パスに存在すること。
2. **入力**: ローカルの設定ファイル（YAML）。Storage ポートでテキストを取得し、ConfigParser で pre-parsed データをドメイン型へパースする。
3. **処理**:
    - 反映方式（全置換 / マージ / 部分更新 / Upsert）に従ってリモートへ反映する。
    - revision を扱うドメインでは、反映直前に現在の revision を取得し、update に渡して楽観的同時実行制御を行う。
    - apply 後、必要なドメインでは運用環境へのデプロイを確認・実行する（`confirmAndDeploy`）。
4. **確認プロンプト**: 破壊的（全置換・削除を伴う）反映の場合、反映対象・削除対象のサマリーを表示し `y/N` で同意を求める。
5. **冪等性**: 同じ設定ファイルでの再実行は安全。最終状態は入力に対して一意。
6. **エラーハンドリング**: 「異常系チェックリスト」に従う。パース失敗は `ValidationError`、通信失敗は `SystemError`、revision 競合は `ConflictError`。apply が複数ステップに分かれる場合、途中失敗時は後続ステップを実行しない（Fail-Fast）。

### capture 系（`<domain> capture`）標準フロー

1. **前提条件**: 認証情報・対象アプリIDが解決済みであること。
2. **入力**: なし（リモートの現在状態を取得）。出力先ファイルパス（デフォルトまたはオプション指定）。
3. **処理**: Configurator ポートでリモートの現在設定を取得し、ConfigSerializer でドメイン型を `Record<string, unknown>` に変換、アプリケーション層で YAML 文字列化して Storage へ保存する。
4. **確認プロンプト / 上書き保護**: 出力先に既存ファイルがある場合は上書き警告。`--force` で抑止可能（「capture / dump の上書き保護」参照）。
5. **冪等性**: 同一のリモート状態に対しては同一内容のファイルを生成（ラウンドトリップ整合）。
6. **エラーハンドリング**: 通信失敗・ファイルIO失敗は `SystemError`。出力先ディレクトリ不在等の入力不備は `ValidationError`。

### diff 系（`<domain> diff`、対応ドメインのみ）標準フロー

1. **前提条件**: 認証情報・対象アプリID・設定ファイルが解決済みであること。
2. **入力**: ローカルの設定ファイル（YAML）。
3. **処理**: ローカル設定をパースし、リモートの現在設定を取得して比較。追加・変更・削除を分類して差分一覧を生成する。
4. **確認プロンプト**: なし（副作用なし）。
5. **冪等性**: 副作用を持たず、リモート・ローカルを変更しない。
6. **エラーハンドリング**: パース失敗は `ValidationError`、通信失敗は `SystemError`。

| ドメイン | 対象 kintone API（取得 / 反映） | 反映方式 | 破壊性 | 前提 | 固有の注意点 |
| --- | --- | --- | --- | --- | --- |
| fieldPermission（フィールドACL） | getFieldAcl / updateFieldAcl | 全置換 | あり（設定外エンティティの権限を削除） | 設定ファイル必須。revision 利用 | apply / capture / diff の3操作。capture は既存ファイル上書き警告 |
| appPermission（アプリACL） | getAppAcl / updateAppAcl | 全置換 | あり（設定外エンティティの権限を削除） | 設定ファイル必須。revision で楽観ロック | `rights` 配列全体を置換。`revision` 省略時は強制上書き。CREATOR/EVERYONE 等の特殊エンティティに注意 |
| recordPermission（レコードACL） | getRecordAcl / updateRecordAcl | 全置換 | あり（設定外の条件・エンティティを削除） | 設定ファイル必須。revision で楽観ロック | 条件（`rights` の条件付きエントリ）ごと全置換 |
| view（一覧/ビュー） | getViews / updateViews | 全置換 | あり（設定外のビューを削除） | 設定ファイル必須。revision で楽観ロック | `revision` 省略時は強制書き込み |
| processManagement（プロセス管理） | getProcessManagement / updateProcessManagement | 全置換 | あり（設定外のステータス・アクションを削除） | 設定ファイル必須。revision で楽観ロック | states キーと actions の from/to の参照整合に注意 |
| generalSettings（一般設定） | getGeneralSettings / updateGeneralSettings | 部分更新（トップレベルキー単位） | 低（指定キーのみ更新） | 設定ファイル必須。revision で楽観ロック | ネスト値オブジェクト（icon / titleField / numberPrecision）は指定時にオブジェクト全体が置換され、子プロパティ単独の部分更新は不可 |

| action（アクション） | getActions / updateActions | 全置換 | あり（設定外のアクションを削除） | 設定ファイル必須。revision で楽観ロック | `actions` マップ全体を置換。destApp.app は環境固有IDのため移行時注意 |
| adminNotes（アプリ管理者用メモ） | getAdminNotes / updateAdminNotes | 全置換（content + includeInTemplateAndDuplicates） | 低（単一リソースの上書き） | 設定ファイル必須。revision で楽観ロック | kintone の AdminNotes 機能そのもの。FormSchema のスキーマ管理とは別概念 |
| plugin（プラグイン） | get-app-plugins / add-app-plugins | `id` ベースのマージ（追加のみ） | なし（削除・無効化は非対応） | 設定ファイル必須 | **特記**: apply は `enabled:true` の未追加 `id` の追加のみ。`enabled:false` は**スキップ＋警告**（kintone は無効状態での追加ができず、追加すると有効化されてしまうため。無効化は UI 手動対応）。無効化・削除に対応する API が存在しないため、無効化・`enabled` 変更の反映・設定外リモートプラグインの削除は非対応（設定外のリモートプラグインは保持される）。diff は `enabled` 変更や deleted も表示するが apply では反映されない（diff と apply の非対称） |

### 補足

- **revision を扱うドメイン**: capture/get 時の revision を update に渡し、楽観的同時実行制御を行う。競合時は `ConflictError`。`revision` 省略は強制上書きとなり競合検出をスキップする。
- **デプロイ**: apply 後の運用環境反映（デプロイ）を伴うドメイン（formSchema/notification/report/action/adminNotes/generalSettings/plugin/processManagement 等）では、apply 後に `confirmAndDeploy` でデプロイを確認・実行する。
- **diff 対応**: 全ドメインが diff を持つわけではない。diff を提供するのは fieldPermission / customization 等、各ドメインの CLI 表に `diff` が明記されたものに限る。
